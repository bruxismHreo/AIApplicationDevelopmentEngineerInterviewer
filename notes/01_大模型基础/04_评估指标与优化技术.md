## 🎯 核心面试题

### Q1: 什么是Perplexity？越低越好吗？

**答案要点：**
- Perplexity衡量模型对测试数据的"困惑程度"
- 通常越低越好，但不是绝对
- 创造性任务需要适度的Perplexity，否则缺乏创意
- 需要警惕过拟合（训练集低，测试集高）

**关键词：** 困惑度、概率分布、过拟合检测

---

### Q2: BLEU和ROUGE有什么区别？

**答案要点：**
- **BLEU**：用于机器翻译，关注精确率（模型输出有多少和参考答案重叠）
- **ROUGE**：用于文本摘要，关注召回率（参考答案有多少被模型覆盖）
- 都是n-gram重叠，但侧重点不同

---

### Q3: Flash Attention的核心思想是什么？

**答案要点：**
- 传统注意力的瓶颈是显存访问，不是计算
- Flash Attention通过分块计算和重新排列顺序，减少显存访问
- 速度提升2-4倍，显存大幅降低
- 不改变算法结果，只优化实现

**关键词：** 显存访问、分块计算、IO感知

---

### Q4: KV Cache的作用是什么？

**答案要点：**
- 缓存已计算过的Key和Value向量
- 避免每次生成都重新计算之前token的attention
- 加速比可达到50倍（生成100个token）
- 代价是额外显存占用（约30-40%）

**关键词：** 缓存、避免重复计算、显存换速度

---

### Q5: 常见的量化方法有哪些？

**答案要点：**

| 量化方法 | 精度 | 显存节省 | 速度 | 精度损失 |
|---------|------|---------|------|---------|
| FP16 | 16位 | 50% | 2x | 几乎无 |
| INT8 | 8位 | 75% | 2-4x | <1% |
| INT4 | 4位 | 87.5% | 4x | 1-3% |

常用方法：GPTQ（INT8）、AWQ（INT4）

## 📝 我的理解（费曼学习法）

**评估指标的核心思想：**

就像考试评分：
- Perplexity = 学生对知识掌握的自信程度（越自信越好，但不能自大）
- BLEU/ROUGE = 答案和标准答案的相似度
- 准确率/召回率 = 做对题的比例

**优化技术的核心思想：**

就像开车：
- Flash Attention = 换更好的引擎（不改变路线，开得更快）
- KV Cache = 记住走过的路（不用重新导航）
- 量化 = 开小车而不是卡车（省油、灵活）
- 剪枝 = 扔掉不必要的行李（轻装上阵）
- 蒸馏 = 跟老司机学习（快速上手）

## 🔑 核心知识点

### 1. Perplexity（困惑度）

**定义：**
```
Perplexity = exp(-1/N × Σ log P(wi|context))

其中：
- N是词的总数
- P(wi|context)是给定上下文下词wi的概率
- 越低表示模型越"自信"
```

**直观理解：**
```
高Perplexity = 模型困惑："下一个词是什么？有太多可能了！"
低Perplexity = 模型确定："下一个词很可能是XX，我很确信！"
```

**为什么不是越低越好？**

| 场景 | 低Perplexity的问题 | 适度Perplexity的优势 |
|------|-------------------|-------------------|
| 创意写作 | 输出保守、千篇一律 | 多样化、有创意 |
| 对话系统 | 回答机械、无个性 | 自然、有趣 |
| 翻译 | 可能牺牲准确性 | 更灵活的表达 |

**面试加分点：**
```
情景分析：
模型A：训练集PPL=5，测试集PPL=50  → 过拟合！
模型B：训练集PPL=10，测试集PPL=12 → 泛化好！

结论：测试集PPL才是关键，训练集PPL低不代表模型好
```

---

### 2. BLEU vs ROUGE

**BLEU Score（Bilingual Evaluation Understudy）**

```
用途：机器翻译评估

计算步骤：
1. 计算n-gram精确率（n=1,2,3,4）
2. 计算短句惩罚
3. 几何平均

公式：
BLEU = BP × exp(Σ wn × log pn)

其中：
- pn是n-gram精确率
- BP是短句惩罚
- wn是权重（通常均匀分布）
```

**ROUGE Score（Recall-Oriented Understudy for Gisting Evaluation）**

```
用途：文本摘要评估

类型：
- ROUGE-N：n-gram召回率
- ROUGE-L：最长公共子序列
- ROUGE-W：加权最长公共子序列
- ROUGE-S：跳跃二元组

核心：关注"参考答案有多少被覆盖"
```

**对比：**

| 指标 | 关注点 | 适用任务 | 计算基础 |
|------|--------|---------|---------|
| **BLEU** | 精确率 | 翻译 | n-gram precision |
| **ROUGE** | 召回率 | 摘要 | n-gram recall |

### 3. Flash Attention

**问题分析：**

```
传统注意力的瓶颈：
- 时间复杂度：O(n²)
- 空间复杂度：O(n²)
- 关键问题：显存访问速度慢（IO-bound）

硬件现实：
- 计算：200 TFLOPS/s（很快）
- 显存带宽：2 TB/s（慢100倍）

结论：加速的关键是减少显存访问！
```

**Flash Attention的解决方案：**

```
核心技巧：
1. 分块计算（Tiling）
   - 把大矩阵分成小块
   - 每次只加载一块到显存
   - 减少显存访问次数

2. 重计算（Recomputation）
   - 前向传播时不存储完整注意力矩阵
   - 反向传播时重新计算
   - 用计算换显存

3. 融合操作（Kernel Fusion）
   - 多个小操作合并成一个大操作
   - 减少kernel启动开销
```

**效果：**

| 指标 | 传统Attention | Flash Attention | 提升 |
|------|--------------|-----------------|------|
| 速度（2048长度） | 1x | 2-3x | 2-3倍 |
| 速度（8192长度） | 1x | 3-4x | 3-4倍 |
| 显存（8192长度） | O(n²) | O(n) | 大幅降低 |
| 最大序列长度 | 2048 | 32768+ | 16倍+ |

---

### 4. KV Cache

**问题：**

```
GPT生成过程：
token1 → 计算attention(1)
token2 → 计算attention(1,2)  ← 重新计算1的attention！
token3 → 计算attention(1,2,3) ← 重新计算1,2的attention！
...

复杂度：O(n²) → 太慢！
```

**KV Cache解决方案：**

```
token1 → 计算K1,V1 → 存入cache
token2 → 从cache读取K1,V1，计算K2,V2 → 存入cache
token3 → 从cache读取K1,V1,K2,V2，计算K3,V3 → 存入cache
...

复杂度：O(n) → 快多了！
```

**显存占用分析：**

```
假设模型参数7B，序列长度2048，隐藏维度4096：

KV Cache大小：
2(层) × 32(头) × 128(头维度) × 2048(长度) × 2字节(FP16)
= 1GB per token
× 2(K和V)
= 2GB

占模型总显存：2GB / 14GB ≈ 14%
```

**实际效果：**

| 序列长度 | 无KV Cache | 有KV Cache | 加速比 |
|---------|-----------|-----------|--------|
| 32 | 1x | 1.2x | 1.2倍 |
| 128 | 1x | 3x | 3倍 |
| 512 | 1x | 8x | 8倍 |
| 2048 | 1x | 25x | 25倍 |

### 5. 量化（Quantization）

**基本原理：**

```
FP32:  [-1.2345678, 2.3456789, ...]  每个参数4字节
  ↓ 量化
INT8:  [-45, 89, ...]                每个参数1字节
  + 缩放因子、零点
```

**量化方法：**

| 方法 | 精度 | 大小 | 速度 | 精度损失 | 适用 |
|------|------|------|------|---------|------|
| FP32 | 32位 | 28GB (7B) | 1x | 0% | 训练 |
| FP16 | 16位 | 14GB | 2x | <0.01% | 训练/推理 |
| BF16 | 16位 | 14GB | 2x | <0.01% | 训练（更稳定）|
| INT8 | 8位 | 7GB | 4x | <1% | 推理 |
| INT4 | 4位 | 3.5GB | 8x | 1-3% | 边缘设备 |

**GPTQ量化（INT8）：**

```
特点：
- 权重量化到INT8
- 激活值保持FP16
- 精度损失<1%
- 量化时间：几小时

命令示例：
python llama.py ./model --wbits 8 --save ./model-8bit
```

**AWQ量化（INT4）：**

```
特点：
- Activation-aware Weight Quantization
- 只量化1%的重要权重
- 精度损失1-3%
- 量化时间：几分钟

优势：
- 速度更快
- 精度更好（相比其他INT4方法）
- 支持多GPU推理
```

### 6. 其他优化技术

**剪枝（Pruning）：**

```
非结构化剪枝：
- 删除单个权重
- 压缩比10:1
- 需要特殊硬件支持

结构化剪枝：
- 删除整个通道/注意力头
- 压缩比2:1
- 不需要特殊硬件
```

**蒸馏（Distillation）：**

```
教师模型（大）→ 学生模型（小）

损失函数：
L = α × L_CE(学生, 真实标签) + (1-α) × L_KL(学生, 教师)

效果：
- 学生模型达到教师90%+性能
- 参数量减少10倍
- 推理速度快10倍
```

**投机采样（Speculative Decoding）：**

```
用小模型快速草稿 → 大模型验证

加速比：2-3倍
条件：小模型有20%+的接受率
```

## 📌 重点记录

### 面试高频考点：

1. **Perplexity的陷阱**
   - 创造性任务需要适度的Perplexity
   - 训练集低、测试集高 = 过拟合
   - 数据分布不匹配也会导致测试集Perplexity高

2. **Flash Attention vs 传统Attention**
   - 传统：关注算法复杂度O(n²)
   - Flash：关注显存访问IO
   - 速度提升2-4倍，不改变结果

3. **KV Cache的trade-off**
   - 显存换速度
   - 显存占用增加30-40%
   - 速度提升20-25倍（长序列）

4. **量化方法选择**
   - FP16：训练和推理通用
   - INT8：生产环境推荐
   - INT4：边缘设备或资源受限

5. **优化技术的组合**
   - Flash Attention + KV Cache：推理加速
   - 量化 + KV Cache：降低显存
   - 蒸馏 + 量化：部署到移动设备

### 常见误区：

❌ 误区1：Perplexity越低模型越好
✅ 正解：创造性任务需要适度的Perplexity，避免输出单调

❌ 误区2：Flash Attention改变了注意力算法
✅ 正解：只改变了实现方式（显存访问模式），结果完全相同

❌ 误区3：量化会大幅降低模型质量
✅ 正解：INT8精度损失<1%，INT4损失1-3%，通常可接受

❌ 误区4：KV Cache总是好的
✅ 正解：显存占用增加30-40%，短序列可能不值得

### 实战建议：

**场景1：云端部署，追求速度**
- Flash Attention + KV Cache + FP16
- 速度提升10-20倍
- 显存需求增加30%

**场景2：边缘设备，显存受限**
- INT4量化 + KV Cache
- 显存降低87.5%
- 精度损失1-3%

**场景3：长文本处理（32K+）**
- Flash Attention 2.0 + KV Cache
- 支持超长序列
- 显存可控

**场景4：移动端部署**
- 蒸馏（7B→1B）+ INT4量化
- 模型大小：3.5GB
- 可在手机上运行
