## 🎯 核心面试题

### Q1: 什么是Mixture of Experts (MoE)？它如何实现参数多但计算快？

**答案要点：**
- MoE将模型分成多个"专家"（Expert），每个专家学习不同知识
- Router负责选择哪些专家参与计算
- 稀疏激活：每次只用1-2个专家，不是所有专家
- 总参数大（如47B），但实际计算量小（如14B）

**关键词：** 稀疏激活、Router、专家专业化

**例子：** Mixtral 8x7B总参数47B，但每次只用2个专家（14B），速度接近13B模型

---

### Q2: 投机采样（Speculative Decoding）的原理是什么？

**答案要点：**
- 小模型快速草拟多个token（实习生）
- 大模型并行验证这些token（老板）
- 接受正确的，拒绝并重新生成错误的
- 加速比取决于接受率（理想情况下3-4倍）

**关键词：** 小模型草拟、大模型验证、接受率

**加速公式：** 加速比 = 接受率 × 小模型加速倍数 / (1 + 小模型加速倍数)

---

### Q3: MoE训练时如何防止专家塌缩？

**答案要点：**
- 专家塌缩：Router总是选同几个专家，其他专家不被使用
- 解决方案：
  1. Expert均衡损失：鼓励均匀使用所有专家
  2. 负载均衡：限制每个专家的样本数量
  3. 噪声注入：给Router加入噪声，增加探索
  4. 专家容量限制：每个专家最多处理N个token

就像专科医院：
- 有心内科、外科、儿科等多个专科医生

就像老板和实习生：
- 实习生快速起草（可能有错）

- 对的保留，错的重新来
- 比老板一个个字写快多了

## 🔑 核心知识点

### 1. Mixture of Experts (MoE)

**基本原理：**

```
传统Dense模型：
每个输入 → 所有参数计算 → 输出

MoE模型：
每个输入 → Router选择 → 少数专家计算 → 加权输出
```

**架构详解：**

```python
# 伪代码
class MoELayer:
    def __init__(self, num_experts=8, top_k=2):
        self.experts = [Expert() for _ in range(num_experts)]
        self.router = Router()
        self.top_k = top_k

    def forward(self, x):
        # 1. Router决定用哪些专家
        expert_scores = self.router(x)  # [batch, num_experts]
        top_k_experts, top_k_indices = expert_scores.topk(self.top_k)

        # 2. 分发到对应专家
        outputs = []
        for i in range(self.top_k):
            expert_output = self.experts[top_k_indices[i]](x)
            outputs.append(expert_output)

        # 3. 加权组合
        final_output = sum(outputs) / self.top_k
        return final_output
```

**关键参数：**

| 参数 | 说明 | 典型值 |
|------|------|--------|
| **num_experts** | 专家数量 | 8, 16, 32 |
| **top_k** | 每次激活几个专家 | 1, 2 |
| **expert_size** | 每个专家的大小 | 7B, 8B |
| **capacity_factor** | 专家容量限制 | 1.0-2.0 |

**实际例子：Mixtral 8x7B**

```
配置：
- 8个专家，每个7B参数
- top_k = 2（每次用2个专家）
- 总参数：46.7B

推理过程：
1. Router输入x
2. 选择Expert3和Expert6
3. 计算这两个专家的输出
4. 加权平均

效果：
- 知识容量：47B级别
- 计算量：14B级别（2×7B）
- 速度：接近13B Dense模型
- 性能：超越LLaMA 34B，接近70B
```

**训练挑战与解决方案：**

**挑战1：专家塌缩（Expert Collapse）**

```
问题：Router总是选同几个专家

原因：某些专家效果更好 → Router倾向于选它们
     → 其他专家得不到训练
     → 效果更差
     → Router更不选它们
     → 恶性循环

解决方案：
1. Expert均衡损失
   loss = task_loss + λ × load_balance_loss

   load_balance_loss = num_experts × Σ (fi × Pi)
   其中fi = 专家i的使用频率
        Pi = 专家i的损失（归一化）

2. 负载均衡策略
   限制每个专家处理的样本数量
   强制均匀分布

3. 专家容量限制
   每个专家最多处理N个token
   超过则丢弃或发送到其他专家
```

**挑战2：显存占用大**

```
问题：需要加载所有专家到显存

解决方案：
1. 专家卸载（Expert Offloading）
   - 不常用的专家卸载到CPU
   - 需要时再加载到GPU

2. 专家共享（Expert Sharing）
   - 不同层共享专家
   - 减少总参数量

3. 分布式推理
   - 不同专家放在不同GPU
   - 多卡并行
```

**挑战3：通信开销**

```
问题：多卡训练时需要传输expert输出

解决方案：
1. All-to-All通信优化
   - 减少通信次数
   - 合并小通信为大连发

2. 本地专家优先
   - 优先使用本卡专家
   - 减少跨卡通信

3. 专家亲和性
   - 相关的数据去同一个专家
   - 减少专家切换
```

### 2. 投机采样（Speculative Decoding）

**基本原理：**

```
传统生成：
输入 "今天天气"
→ 大模型推理 → "真" (100ms)
→ 大模型推理 → "好" (100ms)
→ 大模型推理 → "啊" (100ms)
总时间：300ms

投机采样：
输入 "今天天气"
→ 小模型草拟 "真好啊" (10ms)
→ 大模型并行验证 ["真", "好", "啊"] (100ms)
→ 接受 ["真", "好"]，拒绝 "啊"
→ 大模型重新推理 (100ms)
总时间：210ms（加速约1.5倍）
```

**详细流程：**

```python
# 伪代码
def speculative_sampling(prefix, draft_model, target_model, k=5):
    # 1. 小模型草拟k个token
    draft_tokens = []
    for _ in range(k):
        next_token = draft_model.generate(prefix + draft_tokens)
        draft_tokens.append(next_token)

    # 2. 大模型并行验证
    target_probs = target_model.parallel_forward(prefix + draft_tokens)

    # 3. 逐个接受/拒绝
    accepted_tokens = []
    for i, token in enumerate(draft_tokens):
        draft_prob = draft_model.get_prob(token, prefix + draft_tokens[:i])
        target_prob = target_probs[i]

        # 接受条件
        if target_prob >= draft_prob:
            accepted_tokens.append(token)
        else:
            # 拒绝：从target_model采样
            new_token = sample_from(target_probs[i])
            accepted_tokens.append(new_token)
            break  # 停止接受后续token

    return accepted_tokens
```

**加速分析：**

假设：
- 小模型速度：S = 100 tokens/s
- 大模型速度：T = 10 tokens/s
- 草拟长度：K = 5
- 接受率：A = 70%

**传统方法时间：**
```
生成K个token = K / T = 5 / 10 = 0.5秒
```

**投机采样时间：**
```
小模型草拟K个 = K / S = 5 / 100 = 0.05秒
大模型验证K个 = K / T = 5 / 10 = 0.5秒
总时间 = 0.55秒

有效输出 = K × A = 5 × 0.7 = 3.5个token
加速比 = 3.5 / 0.55 ≈ 6.4倍！
```

**影响加速比的因素：**

| 因素 | 影响 | 优化方法 |
|------|------|---------|
| **接受率** | 越高越快 | 提高小模型质量 |
| **小模型速度** | 越快越快 | 用更小或量化的模型 |
| **草拟长度** | 太长太短都不好 | 动态调整（3-10） |
| **温度** | 太高接受率低 | 保持较低温度（<1.0）|

**接受率优化：**

```
影响接受率的因素：
1. 小模型质量
   - 和大模型同家族（如Llama-2-7B → Llama-2-34B）
   - 用大模型的知识蒸馏得到

2. 任务难度
   - 创意写作：接受率50-70%
   - 事实问答：接受率70-90%
   - 代码生成：接受率60-80%

3. 温度参数
   - 温度0.7：接受率70-80%
   - 温度1.0：接受率50-60%
   - 温度1.5：接受率30-50%

实际策略：
- 自适应草拟长度
- 根据接受率动态调整K
- 如果接受率低，减少K或换小模型
```

### 3. 其他高级优化技术

**LongLoRA（长序列微调）：**

```
问题：
- LoRA难以扩展到32K+序列
- 长序列训练显存需求大

LongLoRA方案：
1. 移位注意力（Shifted Short Attention）
   - 训练时：分块，每块内部注意力
   - 推理时：恢复全局注意力

2. 短序列训练 + 长序列推理
   - 训练：2K序列（低成本）
   - 推理：100K序列（高效率）

3. LoRA优化
   - 只对特定层使用LoRA
   - 减少可训练参数

效果：
- 支持32K-100K上下文
- 训练成本和短序列相同
```

**Sliding Window Attention：**

```
原理：
- 每个token只看前后K个token
- 而不是所有token

Mistral 7B实现：
- 窗口大小：4096
- 滑动窗口：token i 只看 [i-4096, i+4096]
- 支持8K上下文，但计算量减半

优势：
- 计算量：O(n) 而不是 O(n²)
- 显存：线性增长
- 适合超长序列

局限：
- 超出窗口的信息会丢失
- 需要配合其他技术（如KV Cache压缩）
```

**Ring Attention（环形注意力）：**

```
问题：
- 超长序列（1M+）无法放入单卡显存

解决方案：
- 多卡分布式注意力
- 每个卡负责一段序列
- 卡之间环形传递KV

效果：
- 支持1M+上下文
- 线性扩展到多卡
- 常用于长文档处理
```

## 📌 重点记录

### 面试高频考点：

1. **MoE的核心trade-off**
   - 优势：参数多、知识大、计算快（稀疏激活）
   - 代价：显存占用大、训练复杂、通信开销

2. **投机采样的加速条件**
   - 小模型速度要快（至少5-10倍）
   - 接受率要高（>50%才值得）
   - 任务难度适中（太难的接受率低）

3. **MoE vs Dense的选择**
   | 场景 | 推荐方案 |
   |------|---------|
   | 知识密集任务 | MoE（知识容量大）|
   | 资源受限 | Dense（显存小）|
   | 追求极致性能 | MoE（更强）|
   | 简单部署 | Dense（容易）|

4. **专家塌缩的解决方案**
   - Expert均衡损失（核心）
   - 负载均衡策略
   - 噪声注入
   - 专家容量限制

### 常见误区：

❌ 误区1：MoE模型总参数就是计算量
✅ 正解：MoE是稀疏激活，每次只用少数专家

❌ 误区2：投机采样总是加速
✅ 正解：接受率<50%时反而会变慢

❌ 误区3：MoE训练更简单（参数多容量大）
✅ 正解：MoE训练更复杂，需要处理负载均衡、通信等问题

### 实战建议：

**场景1：知识密集应用（问答、代码生成）**
- 推荐MoE架构
- 知识容量大，性能强
- 例子：Mixtral 8x7B

**场景2：边缘设备部署**
- 投机采样 + 量化
- 小模型草拟 + 大模型验证
- INT4量化降低显存

**场景3：长文档处理**
- LongLoRA + Ring Attention
- 支持100K+上下文
- 多卡分布式推理

**场景4：实时对话系统**
- 投机采样 + KV Cache
- 速度提升10-20倍
- 满足实时要求
